# SSO Login Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the wireframe's mobile-number login with a unified, role-agnostic, passwordless sign-in surface — **Continue with Google** (primary) + **email magic-link** (secondary) — keeping the existing teal/marigold pitch panel and the 4 quick-demo shortcuts (re-keyed to email).

**Architecture:** Single-surface login (no per-role tabs). Two new sub-screens — a "Check your email" confirmation (inside the existing `.login-form` panel) and a faked Google OAuth account-picker overlay (modal). All four demo emails route via the existing `ACCOUNTS`/`showApp`/`loginAs` machinery — re-keyed from mobile number to lowercase email. The deep-link handler (`?role=&screen=`) is untouched and continues to bypass auth. Dashboard-design rules apply at the token level (white surfaces, `--shadow-sm`, 4px spacing, Inter type scale, one primary CTA per section) but reuse the existing brand tokens (`--paper`, `--teal-700`, `--marigold`, `--line`) rather than introducing parallel `--gray-*` tokens.

**Tech Stack:** Single-file static HTML/CSS/vanilla-JS prototype (`wireframe/index.html`). No build, no test framework. Frequent commits to `dev`.

**Reference spec:** `docs/superpowers/specs/2026-06-09-sso-login-design.md`

**Verification convention (no test runner):** each task's "failing check" is a Grep confirming the target is absent (for removals) or that the new construct doesn't exist yet (for additions); the "passing check" is the inverse. Task 5 is the manual browser acceptance gate. Commit after each task. All work stays on branch `dev` — do not create/switch branches.

**Subagent flow (per the project's established convention):** for each task below, dispatch in this order — (1) **implementer** subagent does the edits, (2) **spec reviewer** subagent verifies the edits match the spec at `docs/superpowers/specs/2026-06-09-sso-login-design.md`, (3) **quality reviewer** subagent checks for code quality / consistency / no-leftover-old-code. A final (4) **whole-feature reviewer** runs after Task 5 over the full diff.
